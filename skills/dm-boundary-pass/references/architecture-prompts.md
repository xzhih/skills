# 架构讨论提示语

这个文件不是主说明。只有在 `dm-boundary-pass/SKILL.md` 已经触发，并且需要用短句推进讨论时再读。

可直接复用的说法：

```text
先判断：这轮是在谈架构边界、产品形态，还是实现约束。不要混在一起说。
先写清楚：我们希望它成为什么，也不希望它变成什么。
参考别的项目时，先说清楚借哪一点，不要整包照搬。
先分层：这件事属于核心契约、运行时协调、外部能力、宿主应用、安装更新、可选扩展、用户界面，还是持久化存储。
注册不等于授权；授权不等于加载；加载不等于进上下文；进上下文不等于允许执行。
把产品想法先记下来，不要提前写成最终交互。
每个大主题结束时，标记：现在够用了 / 以后再说 / 有冲突再重开。
先做讨论记录，再做主题记录；主题变多后，补减重检查和整体复盘。
长对话要主动做阶段记录，写清当前判断、以后再说的问题、以哪些文档为准、还有哪些问题没解决。
原始资料进 inbox，提炼后的当前结论进 docs，不要混在一起。
每轮复盘都问：我们刚刚是在澄清边界，还是又偷偷设计了一个大系统？
```
