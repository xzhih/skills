# 推进架构讨论

## Overview

这个技能用来把架构讨论收回到真正该定的东西：

```text
边界
分层
责任归属
复杂度放在哪里
哪些现在该定
哪些以后再说
```

它不是实现指南，也不是产品交互设计指南。

## When to Use

这些情况使用：

```text
用户在讨论架构，不是在讨论具体界面
项目开始出现“这一层到底该不该负责这个”的问题
大家开始把产品细节、实现细节、版本计划混着说
需要先确定系统不要成为什么
需要为后续产品讨论打边界
```

这些情况不要用：

```text
用户只是在确认一个产品交互细节
用户要你直接实现
当前问题主要是文档整理，不是边界判断
```

## Core Workflow

按这个顺序推进：

```text
1. 先说清楚这个系统想成为什么，也说清楚不要成为什么
2. 把当前问题分层，不要先谈实现
3. 遇到参考项目时，说清楚借哪一点，不借哪一点
4. 每讨论几个决定，就做一次减重检查
5. 讨论到“现在够用”就先收住，不追求一次定完全部细节
6. 边讨论边落盘，留下主题记录和阶段记录
```

## 分层默认框架

讨论一个新概念时，先判断它更像哪一层：

```text
核心契约
中心协调层
外部能力、集成或扩展
宿主应用、平台壳或部署环境
安装和更新
可选扩展点
用户界面
持久化存储
```

默认规则：

```text
不要先把复杂度压进核心
能放外面、能晚点定、能做成可选能力的，不要过早内建
```

## 必问问题

每个大主题都至少问一遍：

```text
这个复杂度到底该归谁？
这件事更像产品界面，还是更像中心协调层责任？
是不是因为“以后可能有用”才把它放进来了？
这是不是其实属于外部能力、扩展或平台层？
现在不定会不会真的阻塞下一步？
```

## 先拆开的概念

这些东西最容易混在一起：

```text
注册、授权、加载、放进上下文、允许执行
状态、权限、能力、用户看到的展示
引用、生成文件、上下文注入
知识、操作步骤、自动化流程
已安装、可用、本次任务允许使用
系统真实状态、用户界面展示
```

如果概念已经混了，先拆开再继续。

## 测试反向检查

问一个很实在的问题：

```text
不用真实平台、真实外部服务、真实账号、真实模型，
这套架构能不能测试？
```

好的信号：

```text
核心路径可以用假的关键依赖、假的执行器、假的存储、假的事件接收器跑通
```

坏的信号：

```text
必须启动真实平台或真实 UI 环境
必须依赖真实外部服务
必须登录真实账号
必须绑定真实扩展或集成
```

## 落盘要求

至少逐步留下这些东西：

```text
讨论记录
主题记录
阶段记录
后续讨论清单
当前有效结论
```

保存规则：

```text
原始资料放 inbox/ 或类似目录
当前结论放 docs/ 或类似目录
不要把原始资料和当前判断混在一起
```

## Common Mistakes

```text
一上来就讨论实现和接口字段
只说“参考某某项目”，不说具体借什么
没有提前说清楚“不要成为什么”
讨论到一半发现产品、架构、实现已经混成一团
所有东西都想一次定完
只在聊天里说，不持续落盘
```
